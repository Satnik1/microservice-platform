проект, демонстрирующий полный цикл CI/CD для простого Docker-приложения с использованием GitLab CI и Ansible. Автоматически тестирует, собирает и доставляет приложение на тестовый и продакшн серверы.

## Структура

*   `.gitlab-ci.yml`: Описание пайплайна CI/CD.
*   `ansible/`: Плейбуки и конфигурация для деплоя.
    *   `ansible.cfg`: Основные настройки Ansible (отключение проверки ключей, путь к inventory и ключу).
    *   `app_playbook.yml`: Главный плейбук — копирует systemd-юнит, пуллит Docker-образ и запускает сервис.
    *   `app-unit.service.j2`: Шаблон systemd-юнита для управления контейнером приложения.
    *   `inventory/`: Файлы инвентаря.
        *   `test_hosts` и `prod_hosts`: сгенерированные файлы с IP-адресами окружений.

## Пайплайн CI/CD (GitLab CI)

Пайплайн состоит из трех стадий:

1.  **test**:
    *   `test_before_build`: Запускает тесты приложения (через `run-tests.sh`) внутри Docker-контейнера.
    *   `linter_ansible`: Проверяет синтаксис Ansible-плейбука с помощью `ansible-lint`.
2.  **build**:
    *   `build_push_app`: Собирает Docker-образ приложения и пушит его в Docker Hub. Тег образа — короткий SHA коммита.
3.  **deploy**:
    *   `test_app`: Автоматически запускается для веток `master` и `uat`. Деплоит версию приложения (`$CI_COMMIT_SHORT_SHA`) на **тестовый** сервер, используя Ansible с инвентарем `test_hosts`.
    *   `prod_deploy`: Запускается **вручную** только из ветки `master`. Деплоит ту же версию на **продакшн** сервер  с инвентарем `prod_hosts`.

## Как Ansible деплоит приложение

Плейбук `app_playbook.yml` на целевой ВМ делает следующее:
1.  Пуллит Docker-образ `satnik01/skillbox-app:<версия>` с Docker Hub.
2.  Загружает из шаблона `app-unit.service.j2` systemd-юнит, который:
    *   Описывает сервис, запускающий контейнер с пробросом порта `8080:8080`.
    *   Корректно останавливает и удаляет старый контейнер перед запуском нового.
3.  Включает и запускает этот systemd-юнит, гарантируя, что контейнер приложения всегда работает (перезапускается при падении).

## Инвентаризация
*   **`test_hosts` / `prod_hosts`**: Статические файлы, которые, содержат прямые IP-адреса серверов и используются в CI/CD пайплайне.

## Переменные и секреты

Для работы пайплайна в GitLab CI определены переменные:
*   `GIT_LOGIN` / `GIT_PASSWORD`: Для клонирования репозитория с тестами.
*   `DOCKER_USERNAME` / `DOCKER_PASSWORD`: Для пуша образа в Docker Hub.
*   `PRIVATE_VM_KEY`: Приватный SSH-ключ для доступа к целевым ВМ.
*   `IP_ADDRESS`, `APP_VERSION`, `HOST`: Переменные, специфичные для этапов деплоя.
